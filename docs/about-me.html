    Technologies: Veeva's Data Collection platform. I assume (but I never looked it up) that other companies working in the same field have similar platforms and workflows.

    I would rather not talk about this - it wasn't bad, but certainly not something that I would want to do for all my life (or ever again unless necessary, really). It has definitely helped me get used to working in a team, though.

  • 2021 June - November: Application Developer @ Atos.

    Technologies: Linux CLI, C++, Python, Bash, Git, Confluence, Jira.

    The techinal interview went well, and the first couple of weeks as well. However, it became clear to me that mentally I wasn't ready for this job yet.

    Why, you ask?

    Primarily because all my education and side projects (solo and in groups) didn't prepare me to:

    • be alright and feel alright with how much I don't know.
    • ask for help (for some reason I had a subconscious expectation to do everything alone)
    • baby steps (my manager repeatedly reassured me that he was happy with my progress, but I felt that this was completely unwarranted and that I was extremely slow)
    Story time!

    Up until that point, programming meant a few files, maybe a couple of folders' worth of files linked together in a project.

    I remember that one of my first tasks was to track down why updating a certain package caused an error to come up during the post-installation process - but only on every second update. (an error about failing to update the sqlite database)

    It took me some time to recall that we had pre- and post-installation files and that I should look there instead of the source files. Upon finding where the message came from, I had to ask for some help from my manager, something which I was trying to avoid. It didn't take him too long to realise what was going on and then help me understand the problem enough that I was able to fix it.

    Looking back a it, this was a great learning experience. But at that moment all I could think was that I should have been able to solve that alone.

  • 95%

    Technologies: Java

    If you still play the original Doom games you're almost certainly using a sourceport to do so - it's also very likely that you're using GZDoom. Here's the thing, the standard launcher is nice and completely fine if you're only playing Vanilla Doom or not using that many PWADS (Doom mods). Once you start using more PWADS though, especially ones that may cause conflicts if they're loaded in an incorrect order, the simple drag-and-drop method of loading them becomes a bit...annoying.

    So I created WADdup, because I wasn't aware that other launchers existed. I would have created it anyway though, probably. From a technical standpoint, it's not very interesting, just a simple Java desktop application. But it works pretty well, and isn't that all you need in the end?
